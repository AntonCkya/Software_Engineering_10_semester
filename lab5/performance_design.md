# Проектирование производительности: Кэширование и Rate Limiting

## 1. Оценка стратегии кэширования

Проект использует **многоуровневую стратегию кэширования**, реализованную на уровне репозиториев данных. Это является обоснованным выбором, поскольку именно доступ к данным является самым медленным этапом в веб-приложениях.

**Обоснованность выбора:**
- **Точечное кэширование:** Кэшируются только самые частые и тяжелые запросы (например, `get_by_id`, `search_by_name_mask`). Это предотвращает избыточное использование памяти.
- **Конфигурируемость:** Параметры кэша (размер `maxsize` и время жизни `ttl`) задаются через конфигурацию, что позволяет гибко управлять поведением в зависимости от среды (dev, prod).
- **L1 Кэш (LRU):** Используется собственная реализация `AsyncLRUCache` с алгоритмом Least Recently Used (LRU), что эффективно для рабочих нагрузок с паттернами локальности.
- **Кэш на уровне класса:** Кэш является общим для всех экземпляров репозитория, что обеспечивает максимальную эффективность и согласованность.

**Техническая реализация:**
- **`AsyncLRUCache`:** Асинхронный потокобезопасный кэш, реализующий LRU-политику с поддержкой TTL.
- **`make_cached_repository`:** Фабричная функция, которая принимает исходный репозиторий и конфигурацию, и возвращает его кэшированную версию. Это позволяет легко подключать кэширование к любому репозиторию.
- **Ключи:** Генерируются автоматически из имени метода и его аргументов, что обеспечивает правильную адресацию данных.

## 2. Оценка стратегии Rate Limiting

Проект реализует **гибридную стратегию rate limiting**, сочетающую два алгоритма для разных эндпоинтов.

**Корректность проектирования:**
- **Разные политики для разных эндпоинтов:**
  - **`/auth/register` (Sliding Window):** Ограничен 3 запросами в час. Sliding Window — оптимальный выбор для регистрации, так как он точнее предотвращает спам и брутфорс по сравнению с фиксированным окном.
  - **`/auth/login` (Sliding Window):** Ограничен 5 попытками в минуту. Использование sliding window здесь критично, так как он не позволяет "бомбардировать" сервер в момент сброса счетчика.
  - **`/auth/refresh` (Token Bucket):** Пока не реализован в middleware, но класс `RateLimiter` уже поддерживает этот алгоритм.

- **Идентификация по IP:** Используется заголовок `X-Forwarded-For` для получения реального IP клиента за прокси, что является корректной практикой.
- **HTTP-заголовки:** Сервер возвращает стандартные заголовки `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`, что позволяет клиентам адаптироваться.

## 3. Влияние на производительность

Кэширование и rate limiting существенно улучшают производительность и стабильность системы:

- **Кэширование:**
  - **Снижает нагрузку на БД:** Повторяющиеся запросы (например, получение пользователя по ID) обслуживаются из памяти, что уменьшает количество запросов к PostgreSQL.
  - **Уменьшает задержку (latency):** Доступ к данным в памяти происходит в микросекундах, а не в миллисекундах (как при обращении к диску).
  - **Повышает масштабируемость:** Позволяет серверу обрабатывать больше запросов с тем же количеством ресурсов БД.

- **Rate Limiting:**
  - **Предотвращает DoS-атаки:** Ограничивает количество запросов от одного клиента, защищая сервер от перегрузки.
  - **Обеспечивает справедливость:** Гарантирует, что один "плохой" клиент не сможет исчерпать все ресурсы приложения.
  - **Снижает нагрузку на сервер:** Уменьшает количество обрабатываемых запросов на уровне приложения и БД.

## 4. Метрики для мониторинга производительности

Для эффективного отслеживания состояния системы необходимо собирать следующие метрики:

- **Метрики кэширования:**
  - `cache_hit_rate`: Коэффициент попадания в кэш (число попаданий / (число попаданий + число промахов)).
  - `cache_miss_rate`: Коэффициент промаха (обратное к hit rate).
  - `cache_size`: Текущее количество элементов в кэше.
  - `cache_evictions`: Количество вытесненных элементов по политике LRU.

- **Метрики БД:**
  - `db_query_duration_seconds`: Время выполнения запросов к БД (p95, p99).
  - `db_connections_in_use`: Текущее количество используемых соединений с БД.

- **Метрики HTTP-сервера:**
  - `http_request_duration_seconds`: Общее время обработки HTTP-запроса.
  - `http_requests_total{status_code="429"}`: Количество отклоненных запросов из-за rate limiting.

- **Метрики rate limiting:**
  - `rate_limit_requests_allowed_total`: Общее количество разрешенных запросов.
  - `rate_limit_requests_blocked_total`: Общее количество заблокированных запросов.
  - `rate_limit_remaining_requests`: Текущее количество разрешенных запросов для клиента (можно экспортировать).

## 5. Измерение эффективности кэширования (Hit Rate)

**Hit Rate (коэффициент попадания)** — это основной показатель эффективности кэша.

**Расчет:**
```
Hit Rate = (Количество кэш-попаданий) / (Количество кэш-попаданий + Количество кэш-промахов)
```

**Как измерить в данном проекте:**
На данный момент в коде присутствуют отладочные `print`-логи (например, `"{method}: return cached"`), которые позволяют вручную оценить hit rate при тестировании.
